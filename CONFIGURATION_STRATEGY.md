# pyStoOrm - Konfigurationsstrategie

## 🎯 Ziel
Eine einfache, hierarchische Konfiguration, bei der:
- **Defaults vorhanden** sind (Nutzer müssen nicht alles konfigurieren)
- **Nutzer können überschreiben** was sie brauchen
- **Mehrere Projekte** die gleichen Defaults nutzen können
- **Verschiedene Konfigurationsaspekte getrennt** sind (Übersichtlichkeit)

---

## 📁 Dateistruktur

```
pystoorm/
├── config/
│   ├── defaults/
│   │   ├── bootstrap.yml               # 1. Output-Templates, Logging
│   │   ├── naming-conventions.yml      # 2. Naming-Regeln (Primär/Fremdschlüssel, etc.)
│   │   ├── style-guides/
│   │   │   └── python-pep8.yml        # 3. Style Guide (Naming, Formatierung)
│   │   └── attribute-hints.yml         # 4. Default Attribute-Hints (welche Spalte zeigt man?)
│   │
│   └── projects/
│       └── myproject/
│           ├── project.yml             # 5. Project Config (Datenbankverbindung, Overrides)
│           ├── naming-conventions.yml  # (Optional) Locale Naming-Regeln
│           ├── style-guide.yml         # (Optional) Lokale Style-Guide
│           └── attribute-hints.yml     # (Optional) Lokale Attribute-Hints
```

---

## 🔄 Ladeablauf (Hierarchie)

```
1. bootstrap.yml (global defaults)
   ↓
2. naming-conventions.yml (default naming)
   ↓
3. style-guides/python-pep8.yml (default style)
   ↓
4. attribute-hints.yml (default attribute hints)
   ↓
5. project/project.yml (project overrides)
   ↓
6. project/naming-conventions.yml (optional: project naming overrides)
   ↓
7. project/style-guide.yml (optional: project style overrides)
   ↓
8. project/attribute-hints.yml (optional: project attribute hints)
   ↓
FINAL CONFIG (Zusammengeführt mit Defaults)
```

**Wichtig:** Jede Ebene wird **gemergt** (nicht komplett überschrieben).
Wenn nur ein Wert überschrieben werden soll, bleibt der Rest von der vorherigen Ebene.

### 2️⃣ `naming-conventions.yml` - Schlüssel & Spalten

```yaml
# pystoorm/config/defaults/naming-conventions.yml

naming:
  # ===== PRIMÄRSCHLÜSSEL =====
  primary_key:
    pattern: "id|{table_singular}_id"  # Regex: was bedeutet ein PK?
    name_in_code: "{table_singular}_id"  # Wie wird er im Code genannt?

  # ===== FREMDSCHLÜSSEL =====
  foreign_key:
    pattern: "{other_table_singular}_id|{other_table}_id"  # Regex
    name_in_code: "{other_table_singular}_id"

  # ===== FORWARD-SCHLÜSSEL =====
  # (Spalten, die auf ANDERE Tabellen verweisen, aber strukturell nicht FK)
  forward_key:
    pattern: "_ref$|_reference$"
    name_in_code: "{column_base}_ref"

  # ===== TABELLENNAMENSKONVENTIONEN =====
  table:
    # Code: "User", DB: "users"
    singular_form_code: true       # Im Code Singular oder Plural?
    case_in_code: "PascalCase"    # PascalCase, camelCase, snake_case, etc.
    case_in_db: "snake_case"      # Wie sind die Namen in der DB?

  # ===== SPALTENNAMENKONVENTIONEN =====
  column:
    case_in_code: "snake_case"
    case_in_db: "snake_case"

  # ===== SPEZIELLE SPALTEN =====
  special_columns:
    created_at: ["created_at", "createdAt", "creation_date"]  # Muster für Erstellungsdatum
    updated_at: ["updated_at", "updatedAt", "last_modified"]   # Muster für Änderungsdatum
    deleted_at: ["deleted_at", "deletedAt"]                     # Soft-Delete Datum
    version: ["version", "revision"]                            # Versionsspalte
```

**Beispiel: Wie wird das verwendet?**

```python
# Im Coordinator/Generator:
naming_config = config['naming']

# Primary Key Pattern-Matching
if column.name matches naming_config['primary_key']['pattern']:
    print(f"Das ist ein Primärschlüssel!")
    code_name = naming_config['primary_key']['name_in_code']

# Spaltennamen transformieren
original_name = "user_first_name"
case = naming_config['column']['case_in_code']
transformed = transform_case(original_name, case)  # "userFirstName" oder "user_first_name"
```

### 3️⃣ `style-guides/python-pep8.yml` - Code-Stil

```yaml
# pystoorm/config/defaults/style-guides/python-pep8.yml

style:
  # ===== KLASSEN-NAMING =====
  class_naming:
    case: "PascalCase"               # Klassenname-Konvention
    suffix: "Model"                  # Optional: "UserModel" statt "User"
    prefix: ""                       # Optional: "C" → "CUser"

    case: "snake_case"
    get_method_prefix: "get_"        # get_name() statt name()
    set_method_prefix: "set_"        # set_name() statt name=
    is_method_prefix: "is_"          # is_active() für Booleans

  # ===== KONSTANTEN-NAMING =====

    sort_imports: true              # isort-kompatibel
```

### 4️⃣ `attribute-hints.yml` - Welche Spalte zeigt man?

```yaml
# pystoorm/config/defaults/attribute-hints.yml

# "Attitüde": Welche Spalte ist die Hauptspalte einer Tabelle?
# (z.B. für Combobox, Dropdown, Listen-Anzeige)

attribute_hints:
  # Default Heuristiken (probiert in dieser Reihenfolge)
  defaults:
    - name              # "name" Spalte bevorzugt
    - title
    - {table_singular}_name   # "user_name" wenn Tabelle "users"
    - {table_singular}_title
    - description
    - label

  # Tabellen-spezifisch: Welche Spalte zeigen?
  overrides:
    users:
      - last_name      # Bei Users: "last_name" als Hauptspalte
      - first_name
      - email

    products:
      - product_name
      - product_code

    orders:
      - order_number
      - order_date

  # Fallback wenn keine Spalte passt
  fallback:
    - id               # Zeig die ID wenn nichts anderes passt
    - {primary_key}
```

**Verwendungsbeispiel:**

```python
# Im Template oder Generator:
attribute_hints_config = config['attribute_hints']

# Für Tabelle "users"
table_name = "users"
hint = find_attribute_hint(table_name, attribute_hints_config)
# Ergebnis: "last_name" (aus overrides)

# Ausgabe im Repository:
def __str__(self):
    return str(self.last_name)  # Nutzer sieht "Müller" statt "#123"
```

```yaml
  - connection: main_db
    host: 127.0.0.1
    username: myuser
    password: mypassword
    database: myproject_db
    connector: database.mysqlconnector.MysqlConnector
    port: 3306
    # Sicherheit: Passwort auch aus Datei/Umgebungsvariable
    password_from_env: DB_PASSWORD  # oder
    password_from_file: ~/.credentials/db.pass

# ===== PROJEKT-PFADE =====
project_root: /home/user/myproject
output_dir: ${project_root}/generated/orm

# ===== CONFIGURATION OVERRIDES =====
# Hier kann der User Defaults überschreiben:

# Nur einige Tabellen generieren
tables:
  include: ["users", "orders", "products"]  # null = alle
  exclude: ["audit_log", "temp_*"]           # Wildcard möglich

# Lokale Naming-Konventionen (überschreibt Defaults)
naming:
  primary_key:
    pattern: "pk_{table_singular}|{table_singular}_pk"
  foreign_key:
    pattern: "fk_{other_table}|{other_table}_fk"

# Lokaler Style Guide (überschreibt Defaults)
style:
  class_naming:
    suffix: "Entity"  # "UserEntity" statt "UserModel"

# Lokale Attribute Hints (überschreibt Defaults)
attribute_hints:
  defaults:
    - label
    - name
  overrides:
    users:
      - full_name
      - email

# Ausgabe-Templates anpassen
output:
  - from: ./templates/python/model.py.template
    to: ${output_dir}/models/[table].py
    enabled: true

  - from: ./templates/python/repository.py.template
    enabled: false  # Dieses Template nicht generieren
```

## 🔧 Konfiguration laden & mergen

```python
# pystoorm/config/loader.py

class ConfigLoader:
    def __init__(self):
        self.config = {}

    def load_defaults(self):
        """Lade alle Standard-Konfigurationsdateien"""
        self.load_file("config/defaults/bootstrap.yml", "bootstrap")
        self.load_file("config/defaults/naming-conventions.yml", "naming")
        self.load_file("config/defaults/style-guides/python-pep8.yml", "style")
        self.load_file("config/defaults/attribute-hints.yml", "attribute_hints")
        return self

    def load_project(self, project_path):
        """Lade projekt-spezifische Overrides"""
        self.load_file(f"{project_path}/project.yml", "project")

        # Optional: weitere Dateien im Projekt-Verzeichnis
        if os.path.exists(f"{project_path}/naming-conventions.yml"):
            self.merge_deep("naming", self.load_yaml(f"{project_path}/naming-conventions.yml"))

        if os.path.exists(f"{project_path}/style-guide.yml"):
            self.merge_deep("style", self.load_yaml(f"{project_path}/style-guide.yml"))

        if os.path.exists(f"{project_path}/attribute-hints.yml"):
            self.merge_deep("attribute_hints", self.load_yaml(f"{project_path}/attribute-hints.yml"))

        return self

    def load_file(self, path, key):
        """Lade YAML-Datei und speichere unter key"""
        with open(path, 'r') as f:
            data = yaml.safe_load(f)
            self.merge_deep(key, data[key] if key in data else data)

    def merge_deep(self, key, data):
        """Deep-merge: überschreibe nur gegebene Werte"""
        if key not in self.config:
            self.config[key] = {}
        self.config[key] = {**self.config[key], **data}  # Shallow, siehe unten!

    def get_config(self):
        return self.config

# Besser: Rekursives Deep-Merge
def merge_deep(base, override):
    """Rekursives mergen von dicts"""
    for key, value in override.items():
        if key in base and isinstance(base[key], dict) and isinstance(value, dict):
            merge_deep(base[key], value)
        else:
            base[key] = value
    return base
```

## 💡 Anwendungsbeispiele

### Szenario 1: Minimale Konfiguration (nur project.yml)

```yaml
# myproject/project.yml
connections:
  - connection: mydb
    host: localhost
    username: root
    password: secret
    database: myapp
    connector: database.mysqlconnector.MysqlConnector

project_root: /path/to/myproject
```

**Was passiert:**
- Alle Defaults werden geladen
- DB-Verbindung wird gesetzt
- Generiert ORM mit Standard-Naming, Standard-Style, Standard-Attribute-Hints

```yaml
# myproject/project.yml
connections:
  - connection: mydb
    host: localhost
```

## 🎛️ Zugriff auf Konfiguration im Code

```python
# Im Coordinator oder Generator:

class Coordinator:
    def __init__(self, config):
        self.config = config

    def generate(self):
        naming = self.config['naming']
        style = self.config['style']
        hints = self.config['attribute_hints']

        for template in self.config['output']:
            if template['enabled']:  # Neue Option!
                self._render_template(template, naming, style, hints)

    def _render_template(self, template, naming, style, hints):
        # Template-Context mit Konfiguration füllen
        ctx = {
            'naming': naming,
            'style': style,
            'hints': hints,
            'transform_class_name': lambda t: self._transform_class_name(t, style),
            'get_attribute_hint': lambda t: self._get_attribute_hint(t, hints),
        }
        # ... Template rendern ...

    def _transform_class_name(self, table_name, style):
        case = style['class_naming']['case']
        suffix = style['class_naming']['suffix']
        base = transform_case(table_name, case)
        return f"{base}{suffix}"

    def _get_attribute_hint(self, table_name, hints):
        # Finde beste Spalte für diese Tabelle
        if table_name in hints['overrides']:
            patterns = hints['overrides'][table_name]
        else:
            patterns = hints['defaults']

        for pattern in patterns:
            if self._column_exists(table_name, pattern):
                return pattern

        return hints['fallback'][0]  # Fallback
```

## 🚀 Startprozess (Hauptprogramm)

```python
# pystoorm.py

import sys
from config.loader import ConfigLoader

def main():
    project_path = sys.argv[1]

    # 1. Konfiguration laden
    loader = ConfigLoader()
    config = (loader
        .load_defaults()           # Alle Defaults
        .load_project(project_path) # Project-Overrides
        .get_config())

    # 2. Analyzecontroller starten
    controller = AnalyzeController(config)
    schemas = controller.walk()

    # 3. Code-Generator starten
    coordinator = Coordinator(config)
    coordinator.generate(schemas)

if __name__ == "__main__":
    main()
```

## 📊 Zusammenfassung: Hierarchie

```
┌─────────────────────────────────────────┐
│ 1. bootstrap.yml (Output, Logging)      │
├─────────────────────────────────────────┤
│ 2. naming-conventions.yml (Keys, etc)   │
├─────────────────────────────────────────┤
│ 3. style-guides/python-pep8.yml (Code)  │
├─────────────────────────────────────────┤
│ 4. attribute-hints.yml (Display)        │
├─────────────────────────────────────────┤
│ 5. project.yml (DB + ALLE Overrides)    │
├─────────────────────────────────────────┤
│ FINAL CONFIG (Merged + Ready to use)    │
└─────────────────────────────────────────┘
```

Jede Ebene kann von der nächsten überschrieben werden. Nur gegebene Werte werden überschrieben, der Rest bleibt erhalten.

---

## ✨ Vorteile dieser Strategie

✅ **Einfach**: Nutzer muss nur `project.yml` mit DB-Daten füllen
✅ **Flexibel**: Kann alles überschreiben wenn nötig
✅ **Wiederverwendbar**: Mehrere Projekte nutzen gleiche Defaults
✅ **Wartbar**: Defaults und Overrides sind getrennt
✅ **Dokumentiert**: Jede Option hat einen Beispiel-Wert
✅ **Skalierbar**: Neue Konfigurationsoptionen einfach hinzufügbar
